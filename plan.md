# 🗺️ Дорожная карта: Solo Leveling System (Ultimate Edition)

**Видение продукта:** Платформа геймификации с адаптивным ИИ-ядром («Системой»). Приложение превращает рутину в RPG-приключение, где ИИ выступает в роли Гейм-мастера. Пользователь не просто взаимодействует с инструментом — он **живёт** в системе, которая мотивирует через прогресс, историю и социальное признание.

---

## 🧭 Актуальный вердикт и вектор развития (обновлено: 2026-03-29)
**Оценка состояния:** **S-Rank (Potential)** — уже есть “скелет” (архитектура/мультивселенная/атомарность) и “мозг” (ИИ‑сервисы), сейчас проект получает **«душу»** через кинематографичный онбординг и лор.

### Что в проекте уже сильнее рынка
- **Интеллектуальный слой**: античит‑пересчёт наград, адаптивная сложность, генерация контента.
- **Архитектура мультивселенной**: раздельные сейвы по философиям + общий meta‑слой.
- **Иммерсивность**: переход от “форм” к сценам, персонажам и ритуалам.

### Главный риск и противоядие
- **Когнитивная нагрузка** (лора/настроек/систем много) → строгое **Progressive Disclosure**: мощные инструменты (например “Лаборатория”) открываются не сразу.
- **Галлюцинации ИИ** → **жёсткая валидация JSON** и безопасные фолбэки (клиент + Edge Function).

### Приоритет ближайшего цикла (не добавляем новых механик)
Фокус — **Блок 2: “Три Мира” (UI Overhaul)** + **Группировка квестов**, чтобы после “Белой вспышки” пользователь попадал в интерфейс, который **визуально соответствует выбранной философии**.

**Легенда статусов:**
- `[x]` — реализовано (базовый функционал работает).
- `[~]` — есть зачатки / частичная реализация; нужна доработка.
- `[ ]` — запланировано к реализации.

## 📚 Великий Реестр Системы (Great System Registry) — S-Rank контент-слой
*Идея: движок (логика, навигация, ИИ) уже сильный; для удержания на месяцы нужен **системный контент**, который **подстраивается под мультивселенную** (Solo / Mage / Cultivator / Custom), а не разрозненные списки в коде. Реестр — единая точка правды для Мастера, Edge Functions и UI.*

### 1) Скрытые классы (Evolution Path)
Класс не выбирается в начале — игрок **заслуживает** его действиями (теги, метрики, стрики).

- **Solo:** *Теневой Монарх* — например 50 квестов с тегом, связанным с `intelligence`, и 50 с телесным/силовым фокусом; *Повелитель Волков* — N квестов на выносливость (`vitality`) за окно (например 2 недели).
- **Mage:** *Архимаг Пустоты* — накопленные часы фокус-таймера (порог, напр. 100 ч); *Алхимик Разума* — квесты/теги вроде `read` / книги (порог, напр. 20).
- **Cultivator:** *Бессмертный Меча* — длинный стрик тренировок без пропуска (жёсткое правило под Cultivator rules).

*Условия задаются в реестре с привязкой к `SystemId` или шаблону правил.*

### 2) Титулы (пассивные модификаторы)
Титул = не только строка, а **пассив** (через `TitleDef.effects` / `Buff` / множители), согласованный с экономикой.

- Примеры концепций: после серии штрафных зон — титул с **+% к золоту**; 30 дней без провалов — **смягчение штрафов** за просрочку; N записей в дневнике мира — **скидка на ИИ-данжи** / дешевле токены генерации.
- Реализация уже частично есть (`titles_data`, `TitleDef`); реестр должен **централизовать** условия разблокировки и эффекты по философиям.

### 3) Предметы и лут (Economy Sinks)
- **Расходники:** эликсир восстановления стрика; свиток смены философии **без потери уровня** (дорого, с ограничениями); зелья отката и т.д.
- **Артефакты (экипировка):** кольцо с бонусом к статам/промпту при ИИ-квесте; чётки — ускорение награды медитации и т.п.
- Каталог расширяется в реестре; цены — через `EconomyScale` / будущие `ValuePoints`.

### 4) Достижения (Milestones)
Смесь открытых и скрытых целей: онбординг без скипов, первый «аномальный» данж / Кровавые Врата, порог вклада в **Мировые Врата** (напр. 1000 единиц), социальные вехи.

### 5) События и квесты (Live Ops / динамика)
Триггеры, которые ИИ или планировщик вбрасывает в ленту (с учётом `SystemRules`):

- **День Очищения** (напр. воскресенье): квесты на порядок/уборку — х2 опыта.
- **Магическая буря:** 24 ч — усиление ветки интеллекта, обязательность, повышенный лут.
- **Испытание Небес (Cultivator):** жёсткий дедлайн на ключевых уровнях; провал — регресс уровня (только если согласовано с балансом и этикой продукта).

### 🛠 Техническая реализация (без раздувания `main.dart`)
Папка **`lib/src/core/content/`** (или поэтапно перенос из `lib/src/data/`):

| Файл (план) | Назначение |
|-------------|------------|
| `achievements_registry.dart` | Список `AchievementDef`, условия, l10n-ключи |
| `titles_registry.dart` | `TitleDef` + эффекты / привязка к `Buff` |
| `items_registry.dart` | Расходники и артефакты для лавки/дропа |
| `hidden_classes_registry.dart` | Условия эволюции + привязка к `SystemId` |
| (опц.) `live_events_registry.dart` | Шаблоны ивентов, cron-подобные правила |
| (опц.) `content_catalog.dart` | Единая точка экспорта / поиск по id для ИИ |

**Главная фишка для ИИ:** промпт Мастера / Edge Function получает **сжатое описание реестра** (id, триггеры, теги): при разборе ввода «сегодня много читал» система проверяет пороги тегов и может предложить выдачу титула/ивента (**строгая валидация на клиенте/сервере**, без произвольных наград).

### Приоритет наполнения (рекомендация)

2. Достижения и предметы-sinks — удержание и экономика.
3. Live Ops — после стабильного реестра и античита.

## 💭 Заметки, идеи и открытые вопросы

*Сюда можно добавлять размышления при ревью плана — не часть жёсткой спецификации, а рабочие гипотезы.*

1. **Приоритизация:** Фазы 0 (безопасность ключей, фон, тесты) и Фаза 4 (пуши) сильно влияют на доверие и удержание, но не «видны» в демо. Имеет смысл явно выделить *релизные вехи* (например, «MVP+» = ключи + минимальные тесты опыта/квеста).

2. **ИИ-балансировщик vs ручная форма:** Сейчас игрок задаёт награды вручную. Переход к «только название + суть» потребует жёсткой валидации JSON от модели, откатов при ошибке и, возможно, **потолков** по уровню охотника — иначе античит останется обходимым.

3. **Штрафная зона и дебаффы:** Нужна единая модель баффов/дебаффов в Hive (тип, источник, `expiresAt`), иначе красный штрафной квест будет «особым случаем» в коде квестов.

4. **Конфликты облака:** LWW проще, но для RPG с квестами лучше документировать **правила слияния** (например: квесты по `id`, охотник по `updatedAt`, инвентарь — осторожно с дублями).

5. **Социальный слой:** Таблицы лидеров и друзья потребуют бэкенда шире, чем текущий бэкап (`game_backups`). Стоит заранее решить: расширять Supabase-схему или отдельный сервис.

6. **Нарратив:** Сюжетные `story`-квесты на уровнях хорошо стыкуются с уже заведённым типом квеста в модели — остаётся контент, триггеры разблокировки и UI «вехи», чтобы не было просто ещё одной строки в списке.

7. **Профиль и «AAA»-UX:** Ранг E–S, титулы, компактный опыт и неоновые карточки уже задают тон; дальше радар/heatmap (Фаза 1) логично reuse те же токены темы и шрифты цифр, чтобы не плодить второй визуальный язык.

8. **Выбор наставника + TTS (идея на будущее):** добавить экран выбора “Мастера” (не только арт, но и голос/манера речи). Дальше можно подключить TTS, чтобы реплики системы озвучивались (с переключателем и кешированием аудио), а философии получали реально разные “голоса”.

Если появятся новые вопросы по приоритетам или формулировкам чеклиста — дописывайте прямо в этом разделе или правьте фазы выше.
